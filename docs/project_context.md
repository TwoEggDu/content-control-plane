# Project Context

## 产品名称

- 中文：内容控制平台
- English: Content Control Plane

## 为什么做这个系统

这个仓库不是凭空起题，而是把你过去长期做过的几类工作，抽象成一个更完整、更可展示、也更容易持续演进的平台型项目。

过去的工作主线已经很明确：

- 你长期覆盖 Unity 工具链、资源治理、构建发布、多平台交付和部分质量门禁
- 你做过的能力不只是“编辑器小工具”，而是把资源检查、发布链路、版本控制、规范治理和协作流程串起来
- 你已经在往“项目内工具 -> 跨项目平台 -> 技术负责人”这条路径上推进

这个系统的价值，在于把这些分散在 Unity 工具、CI、脚本、聊天通知和人工协作里的能力，提升成一个统一的控制面。

## 来自既有仓库的背景

### 来自 `GameEngineDev`

`GameEngineDev` 说明了这个项目背后的更高层目标：

- 你的主线定位不是普通业务开发，而是客户端基础架构、工具链、交付治理和资源成本治理
- 你真正值钱的地方，是把项目内交付问题抽象成跨项目可复用的平台能力
- “从项目内工具 -> 跨项目平台”已经是你明确要强化的叙事主线

### 来自 `HT`

`HT` 说明了你过去已经做过的能力边界：

- 构建发布和多平台出包
- 资源导入规则和资产格式治理
- 资源检查、Missing 检测、报告输出和责任归属
- 发布审批、灰度、回滚、日志和质量域平台的长期规划

其中最直接的参考，是 PX 项目已经存在一套成熟的资源检查与报告框架，说明“规则化检查 + 报告输出 + CI 集成 + 通知”这条线并不是概念设计，而是你真实做过的事情。

## 这个仓库的定位

这个仓库应当是一个产品代码仓库，而不是纯研究仓库或想法仓库。

它承载的不是扫描器本身，而是扫描结果导入、问题跟踪、状态流转、资源归属、发布控制和审计支撑这类控制面能力。

更具体地说：

- Unity 检查器、命令行扫描器或 CI 继续负责产生检查结果
- 本仓库负责接入、展示、查询、流转、统计、审批和审计
- 第一阶段先建立“质量门禁控制面”的闭环
- 第二阶段再逐步补齐资源发布控制、审批和回滚等能力

## 为什么现在值得做

这个项目同时满足三个目标：

1. 它能承接你过去做过的真实能力，而不是重新虚构一个方向。
2. 它能把“资源治理 + 发布治理”从项目经验上升为平台设计作品。
3. 它适合作为 Go 后端样本，补足你在服务端控制面方向的表达。

## 当前推荐的仓库职责

这个仓库当前应优先承担以下职责：

- 作为内容控制平台的主仓库
- 用文档先固定产品边界、开发流程和决策协议
- 再用 SDD 和 ADR 驱动后续实现
- 再通过小闭环方式推进可运行的 Go 服务

## 当前不承担的职责

以下内容当前不应塞进这个仓库作为第一优先级：

- Unity 扫描器和检查规则本体
- 资源分发链路、CDN 或补丁下发系统
- 大而全的性能平台、日志平台或设备平台
- 过早的微服务拆分

## 当前结论

这个项目本质上是在做一个面向游戏研发生产链路的内容控制面。

它既是产品项目，也是你过去工具链和治理经验的平台化表达。后续的 `product_intent`、SDD 和实现，都应围绕这个定位展开。
