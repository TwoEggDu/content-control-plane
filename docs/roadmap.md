# Roadmap

## 文档目的

本文档用于定义这个项目的阶段目标和推进顺序。

它不是详细任务清单，也不是日历排期。它回答的是：当前先做哪一段闭环，后面再扩什么。

## 当前阶段判断

当前仓库仍处在早期阶段，推荐按能力闭环而不是按模块数量推进。

## 阶段划分

### M0 文档与流程基线

目标：先把项目为什么做、怎么推进、哪些问题需要拍板固定下来。

完成标准：

- `project_context`
- `product_intent`
- `autonomous_workflow`
- `owner_decisions/current`
- 文档入口和阅读路径

状态：`IN_PROGRESS`

### M1 质量门禁控制面最小闭环

目标：打通第一个真正可运行的后端闭环。

范围：

- 扫描结果导入
- 扫描任务记录
- 问题列表与详情查询
- 基础状态流转
- 附件和报告关联

完成标准：

- 能导入一份样例扫描结果
- 能查询扫描任务和问题列表
- 能查看问题详情
- 能完成至少一条状态变更

前置依赖：

- `D-001` 和 `D-002` 决策确认
- 第一版 SDD 拆分完成

状态：`PLANNED`

### M2 问题治理与统计增强

目标：把质量门禁从“可导入可查看”升级到“可持续治理”。

范围：

- 问题批量处理
- 问题操作日志
- 趋势统计
- 通知和定时任务的最小版本

状态：`PLANNED`

### M3 发布控制第一版

目标：在质量门禁闭环稳定后，补发布控制面。

范围：

- 资源版本登记
- 环境提升
- 发布审批
- 灰度和回滚记录
- 发布审计链路

状态：`PLANNED`

## 当前推荐策略

- 不把质量门禁和发布控制同时作为第一实现目标
- 不按“大平台外壳”推进
- 先交付最小闭环，再进入统计、通知和发布治理

## 何时更新这份文档

满足以下任一条件时应更新：

- MVP 范围变化
- 阶段顺序变化
- 某个阶段完成标准变化
- 新增一个足够独立的里程碑阶段
