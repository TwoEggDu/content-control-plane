# Roadmap

## 文档目的

本文档用于定义这个项目的阶段目标和推进顺序。

它不是详细任务清单，也不是日历排期。它回答的是：当前先做哪一段闭环，后面再扩什么。

## 当前阶段判断

当前仓库已经从纯文档阶段进入“可运行 MVP 基线”阶段。

推荐继续按能力闭环推进，而不是提前扩模块数量。

## 阶段划分

### M0 文档与流程基线

目标：先把项目为什么做、怎么推进、哪些问题需要拍板固定下来。

完成标准：

- `project_context`
- `product_intent`
- `autonomous_workflow`
- `owner_decisions/current`
- 文档入口和阅读路径

状态：`DONE`

### M1 质量门禁控制面最小闭环

目标：打通第一个真正可运行的后端闭环。

范围：

- 扫描结果导入
- 扫描任务记录
- 问题列表与详情查询
- 基础状态流转
- 附件和报告关联

当前已完成：

- 第一版 SDD 主体
- 第一版数据模型和 MVP API 契约
- `Gin` API 骨架
- 内存仓储版本的导入到查询闭环
- 应用层基础测试

剩余工作：

- PostgreSQL 持久化
- migration
- HTTP 集成测试
- 更完整的本地运行手册

状态：`IN_PROGRESS`

### M2 问题治理与统计增强

目标：把质量门禁从“可导入可查看”升级到“可持续治理”。

范围：

- 问题批量处理
- 问题操作日志增强
- 趋势统计
- 通知和定时任务的最小版本

状态：`PLANNED`

### M3 发布控制第一版

目标：在质量门禁闭环稳定后，补发布控制面。

范围：

- 资源版本登记
- 环境提升
- 发布审批
- 灰度和回滚记录
- 发布审计链路

状态：`PLANNED`

## 当前推荐策略

- 不把质量门禁和发布控制同时作为第一实现目标
- 不按“大平台外壳”推进
- 先把 M1 做成真实可持久化后端，再进入统计、通知和发布治理

## 当前开发计划

接下来默认按下面顺序推进：

1. `PostgreSQL + migration`
2. `Repository` 从内存实现切换到数据库实现
3. HTTP 集成测试和导入冒烟测试
4. 对象存储真实接入
5. 再进入统计和问题治理增强

## 何时更新这份文档

满足以下任一条件时应更新：

- MVP 范围变化
- 阶段顺序变化
- 某个阶段完成标准变化
- 新增一个足够独立的里程碑阶段
