# ADR-0001：采用文档驱动的项目推进方式

**日期**：2026-03-11
**状态**：ACCEPTED
**决策者**：项目维护者

## Context

这个仓库当前仍处在产品边界和系统设计快速收敛阶段。

如果直接进入代码实现，会出现几个问题：

- 产品目标、MVP 范围和第一版交付形态容易反复摇摆
- 系统设计会被零散记录在 README、聊天和代码注释里，后续很难维护
- 真正需要维护者拍板的事项，容易和普通执行问题混在一起
- 单人推进时，执行细节和方向决策容易失去边界

同时，这个项目本身也承担“平台化表达”和“作品集”职责，需要让外部读者能看清：

- 为什么做
- 先做什么
- 为什么这么设计
- 当前走到哪一步

## Decision

我们决定在项目早期采用文档驱动的推进方式，并固定以下结构：

- `project_context`：解释项目背景和仓库定位
- `product_intent`：定义产品边界和 MVP 范围
- `owner_decisions/current`：集中管理待拍板事项
- `roadmap`：定义阶段顺序
- `backlog`：维护当前工作项池
- `sdd/`：承载系统设计
- `adr/`：记录关键决策
- `testing_strategy` 与 `runbook`：承接交付和运行

默认推进规则是：

- 日常可逆事项直接推进
- 真正影响边界、核心模型或交付形态的事项再上提决策
- 当设计稳定后，再进入骨架实现和可运行闭环

## Consequences

### 正面后果

- 项目结构更清楚，后续扩展时不容易把所有信息堆进一个文件
- 维护者只需要处理少量高价值决策，而不必跟踪所有执行细节
- 更适合作为对外展示的工程化项目，而不只是内部草稿
- 后续进入实现阶段时，任务拆分和设计演进会更稳

### 负面后果

- 前期文档投入会增加，短期看起来比直接写代码更慢
- 如果文档不持续更新，容易出现文档和实际进展脱节
- 需要持续约束“什么写 backlog，什么写 ADR，什么写 owner decisions”

### 中性后果

- README 不再承担所有信息入口的职责，而是变成索引页
- 一部分设计内容会先以轻量文档存在，后续再并入正式 SDD
