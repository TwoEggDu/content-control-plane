# ADR-0002：第一版 Web 框架使用 Gin

**日期**：2026-03-11
**状态**：ACCEPTED
**决策者**：项目维护者

## Context

这个项目的第一版需要尽快进入一个可运行的控制面后端闭环。

在 Web 框架选择上，之前文档里曾默认写成 `chi`，但当前维护者已经明确要求第一版使用 `Gin`。

这个选择会影响：

- API 层的组织方式
- 中间件使用方式
- 参数绑定与校验习惯
- 后续最小后台页面或管理接口的接入方式

同时，这个项目虽然选择了具体 Web 框架，但仍然需要保持领域层和存储层不被框架侵入。

## Decision

第一版 Web 框架使用 `Gin`。

具体约束如下：

- `Gin` 只负责 HTTP 路由、请求绑定、中间件和响应输出
- 业务逻辑放在应用层和领域层，不直接绑死在 `Gin Handler` 中
- 持久化层不依赖 `Gin`
- 后续即使替换 Web 层，也不应影响核心问题模型、状态流转和数据访问逻辑

## Consequences

### 正面后果

- `Gin` 生态成熟，更容易快速搭起第一版 API 服务
- 请求绑定、中间件、路由组织和基础扩展能力更完整
- 更适合在第一版阶段提高实现速度，减少基础 Web 层阻力

### 负面后果

- 相比更轻的框架，`Gin` 的框架痕迹会更强
- 如果约束不好，容易把请求上下文和绑定逻辑渗透到业务层

### 中性后果

- 这个决策只固定 Web 层，不改变“Go 模块化单体”的总体架构
- 后续仍需在 SDD 中明确 `handler -> application -> domain -> repository` 的边界
