# 2026-03-11 待决策记录

## D-001 第一版实现范围

- 决策问题：第一版是否严格收敛到“质量门禁控制面闭环”，把发布控制保留到后续阶段？
- 为什么重要：
  - 这会直接影响 `product_intent`、SDD 范围、表设计和开发节奏
  - 如果第一版同时做质量门禁和发布控制，前期范围容易明显膨胀
- 推荐方案：
  - 第一版严格收敛到质量门禁闭环
  - 发布控制保留在产品意图和 SDD 草图中，但不进入最早实现闭环
- 可选替代方案：
  - 同步落最小发布控制模型，例如 `release`、`approval`、`rollback_record`
- 决策结论：
  - 选择推荐方案
  - 第一版仅实现扫描导入、问题查询、状态流转、附件和基础统计
  - 发布控制保留在文档设计和后续阶段，不进入第一版实现范围
- 延迟决策的影响：
  - `product_intent` 和 SDD 的范围会一直摇摆
  - 项目骨架和数据库设计容易反复调整
- 当前状态：`DECIDED`

## D-002 第一可运行交付形态

- 决策问题：第一可运行版本应优先做 API + 导入链路，还是直接做带最小页面的管理后台？
- 为什么重要：
  - 它决定第一轮实现重心是领域与数据闭环，还是同时处理页面和交互
  - 它会影响项目骨架、测试方式和开发顺序
- 推荐方案：
  - 先 API 优先
  - 第一轮先打通扫描导入、扫描任务查询、问题查询和状态流转
- 可选替代方案：
  - 直接做服务端渲染的最小管理页面，方便更早展示
- 决策结论：
  - 选择推荐方案
  - 第一可运行版本优先交付 API 和导入链路
  - 先打通扫描导入、扫描任务查询、问题查询和状态流转的后端闭环，再决定是否补最小管理页面
- 延迟决策的影响：
  - 会导致第一轮骨架无法明确是“后端优先”还是“完整控制面优先”
- 当前状态：`DECIDED`
