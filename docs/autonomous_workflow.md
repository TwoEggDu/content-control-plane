# 自主协作工作流

## 目的

本文档用于定义这个仓库中的默认开发流程，使项目可以持续推进，同时把真正需要你拍板的事项压缩到少量关键问题。

## 默认工作模式

助手默认应以自主推进的方式工作。

这意味着：

- 日常的梳理、设计、脚手架和实现，不需要逐项等待批准
- 在需求尚未完全稳定时，优先选择可回退、可修正的方案
- 不因为可以后续再细化的小问题停住
- 只有当某个决定会实质影响产品边界、数据模型或长期实现成本时，才停止并上提

## 你的职责

你不需要管理日常执行细节。

你主要只需要做这些事：

- 按固定节奏查看 `docs/owner_decisions/current.md`
- 如有需要，再查看当天日期文件，例如 `docs/owner_decisions/2026-03-11.md`
- 对开放的产品或架构问题做决定
- 在必要时覆盖默认推荐方案
- 当 MVP 边界发生明显变化时确认是否接受

## 助手的职责

助手应当：

- 在开始较大工作前阅读 `AGENTS.md`、`README.md`、`docs/project_context.md`、`docs/product_intent.md`、`docs/sdd/README.md` 和 `docs/owner_decisions/current.md`
- 默认持续推进项目，而不是等待常规批准
- 只把真正阻塞推进的事项写入 `docs/owner_decisions/` 目录
- 对每个待决策事项给出明确推荐，而不只是抛问题
- 让待决策事项保持简短、清晰、高信号
- 在每个阶段结束后同步更新相关文档，而不是让文档与实现脱节

## 什么情况需要上提给你

满足以下任一条件时，应把事项上提为待你决策：

- 会改变 MVP 范围
- 会固定之后会广泛传播的核心术语
- 会影响核心数据模型或状态流转设计
- 会改变第一版交付形态
- 如果假设错误，会带来明显的产品风险或返工成本

## 什么情况不需要上提

以下事项不应频繁打断你：

- 目录命名
- 文档措辞优化
- 可回退的小规模重构
- 不破坏兼容的技术默认值
- 本地开发配置的细节
- 符合当前默认路线的首轮脚手架工作

## 决策队列协议

所有面向你的待决策事项都集中放在 `docs/owner_decisions/` 目录中。

目录约定如下：

- `current.md`：当前仍未决的总表
- `YYYY-MM-DD.md`：当天新增事项
- `README.md`：目录说明和使用方式

每个开放事项至少应包含：

- 一个 ID
- 决策问题本身
- 为什么重要
- 推荐方案
- 可选替代方案
- 延迟决策的影响
- 当前状态

## 默认开发顺序

当前仓库的推荐顺序不是“先写很多代码”，而是先把设计和推进机制固定下来：

1. 先澄清 `project_context`
2. 再固定 `product_intent`
3. 再拆出 `docs/sdd/` 下的系统设计
4. 再用 ADR 记录关键不可逆决策
5. 再按小闭环方式实现和验证

## 当前阶段的实现节奏

结合当前项目状态，默认按下面顺序推进：

1. 完成文档入口、开发流程和待决策机制
2. 明确第一阶段产品意图和 MVP 边界
3. 落第一版 SDD，总结领域模型、接入链路和运行形态
4. 搭 Go 项目骨架和基础运行方式
5. 打通“导入扫描结果 -> 查询扫描任务 -> 查询问题列表 -> 查看问题详情”
6. 补充问题状态流转、操作日志和附件
7. 再进入统计、通知和发布控制

## 默认推进原则

如果某个事项不在决策队列中，助手应默认认为它可以安全推进。

如果某个决定重要但仍然可逆，助手应：

- 选择最保守但可工作的方案
- 在相关文档中记录这个假设
- 持续推进，直到出现真正阻塞为止

## 每个迭代后的最小更新要求

每完成一个阶段，至少应更新这些内容：

- `README.md` 中的入口和当前状态
- 相关设计文档
- `docs/owner_decisions/current.md`

如果出现新的关键待决策事项，还应新增当天日期文件并同步汇总到 `current.md`。

## 当前结论

这个仓库从现在开始采用“文档先行 + 决策集中上提 + 小闭环实现”的推进方式。

下一阶段默认先把 `product_intent` 和 SDD 继续补齐，再开始真正的项目骨架实现。
