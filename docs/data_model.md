# Data Model

## 文档目的

本文档用于描述这个系统的核心数据对象、建模原则和当前实现优先级。

它比 `architecture.md` 里的草图更聚焦“对象和关系”，但还不是最终数据库 DDL。

## 这份文档和其他文档的关系

- `product_intent`：回答为什么需要这些对象
- `SDD`：回答这些对象如何进入模块和接口
- 本文档：回答系统当前有哪些一等公民对象，以及它们为什么这样建模

## 当前一等公民对象

### `project`

代表一个被平台管理的游戏项目或内容域。

作用：

- 作为扫描任务、资源、规则和问题的归属边界
- 作为权限和统计的第一层过滤维度

### `scan_task`

代表一次扫描结果导入行为。

作用：

- 记录来源、分支、提交号、扫描器版本、耗时和导入状态
- 作为问题、附件和报告的批次归属

### `resource_item`

代表被扫描和治理的资源对象。

作用：

- 承载资源路径、GUID、类型、模块和责任人
- 作为问题聚合、归属和统计的基础

### `issue_item`

代表一次可追踪的问题实例。

作用：

- 承载规则命中结果、严重级别、状态和责任人
- 连接扫描任务、资源对象和规则定义

### `issue_action_log`

代表一次问题上的状态或处理动作。

作用：

- 记录谁在什么时候做了什么动作
- 支撑审计、复盘和协作追踪

### `attachment_file`

代表与扫描任务相关的报告、截图、日志或原始结果附件。

作用：

- 保存报告产物的元信息
- 与对象存储打通，而不是把大文件塞进数据库

## 当前辅助对象

### `rule_def`

第一版可能先弱化实现，但概念上仍然存在。

作用：

- 给问题挂上规则编码、分类和严重级别
- 未来支持项目级规则启停和归档

### `user_account`

第一版可先保持极简，但后续一定会进入核心模型。

作用：

- 负责责任人、操作人、审批人和权限边界

## 当前建模原则

### 1. 导入记录必须可追溯

任何问题都应该能回到它来自哪次扫描、哪份输入、哪组附件。

### 2. 问题实例和资源对象要分开

`issue_item` 是问题事实，`resource_item` 是被管理对象。两者不能混成一张表。

### 3. 状态变化必须保留历史

不要只保留当前状态，还要保留动作日志。

### 4. 大文件走对象存储，数据库只存元信息

报告、截图、原始产物都不应成为关系库的负担。

### 5. 第一版优先稳定主键和归属关系

早期最容易返工的不是字段数量，而是对象边界和关系方向。

## 第一版实现优先级

推荐先稳定这些对象：

1. `project`
2. `scan_task`
3. `resource_item`
4. `issue_item`
5. `attachment_file`
6. `issue_action_log`

## 当前开放问题

这些问题暂时不在本文件内拍板，后续可进入 `owner_decisions` 或 SDD：

- 问题指纹的最小生成规则
- `issue_item` 是否要区分“问题实例”和“问题族”
- 责任人是直接挂在问题上，还是优先从资源归属推导
- 附件是只挂扫描任务，还是允许后续挂问题级附件
