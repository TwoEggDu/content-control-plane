# Product Intent

## 1. 文档目的

本文档用于明确“内容控制平台”的产品意图、目标用户、问题边界、MVP 范围和成功标准。

它不是详细需求文档，也不是技术设计文档。它的作用是：

- 为后续 `docs/sdd/` 下的系统设计提供产品边界
- 为架构决策提供背景
- 为实现阶段提供“做什么 / 不做什么”的判断依据

## 2. 产品定义

内容控制平台是一个面向游戏项目研发生产链路的内部控制面。

它的目标不是取代 Unity 检查器、命令行扫描器或 CI，也不是直接承担资源分发，而是把这些链路产生的结果和操作，汇总成一个统一的可见化、可追踪、可控制的平台。

系统当前聚焦三类能力：

- 资源质量门禁
- 问题跟踪与状态流转
- 资源发布控制与运维支撑

## 3. 核心问题

这个系统想解决的，不是“怎么写更多检查规则”，而是“怎么让内容质量和发布过程变得可控”。

当前问题主要体现在：

- 扫描结果分散在 Unity 工具、日志、CI 和群消息中，缺少统一入口
- 问题责任人、处理状态和历史流转不清楚
- 资源版本、审批、灰度和回滚信息容易散落在表格、脚本和人工流程中
- 缺少按项目、规则、资源、负责人和时间维度的趋势观察
- 关键动作缺少审计链路，后续复盘困难

## 4. 目标用户

### Primary Users

第一阶段的核心用户是内部技术和交付角色：

- 工具链 / 内容治理负责人
- 客户端基础架构或技术负责人
- 资源检查和发布流程的维护者
- 需要查看问题、审批发布、跟踪状态的项目核心成员

### Future Users

后续可以扩展到：

- 更完整的发布运维角色
- 跨项目治理人员
- 更细粒度的资源责任人协作场景

但这些不属于第一阶段必须完成的范围。

## 5. 产品目标

第一阶段产品需要先帮助团队回答这些问题：

- 这次扫描到底发现了什么问题
- 哪些资源、规则和责任人最值得优先关注
- 问题现在处于什么状态，最近是谁改动过
- 哪些问题是新出现的，哪些已经解决，哪些被忽略
- 与某次扫描或某个资源相关的报告和附件在哪里

在后续阶段，系统再进一步回答：

- 哪些资源在什么时候被发布到什么环境
- 谁审批过，谁执行过，是否存在灰度或回滚记录

## 6. 产品原则

### 6.1 控制面优先，不搬运扫描逻辑

第一阶段重点在结果接入、状态流转、查询展示和发布治理，而不是把 Unity 侧检查器迁进后台。

### 6.2 可追踪性优先于炫技

系统价值首先来自幂等导入、问题指纹、责任归属、历史记录和审计链路，而不是复杂架构。

### 6.3 小闭环优先于大平台外壳

先打通一个真实可运行的闭环，再扩展更多域能力。

### 6.4 平台化表达要贴近真实经验

后续设计应尽量承接你过去做过的资源门禁、报告输出、发布治理和跨项目平台化经验，而不是为了“像平台”而额外堆概念。

## 7. 当前 MVP 范围

当前默认推荐的 MVP 范围是“质量门禁控制面闭环”：

- 导入扫描结果
- 记录扫描任务
- 展示问题列表和问题详情
- 支持责任人分配、忽略、关闭、验证和状态流转
- 支持基础趋势统计
- 支持附件和报告关联

发布控制相关能力在第一阶段保留方向和数据模型草图，但不默认作为最早实现闭环。

## 8. 非目标

第一阶段不以这些能力为目标：

- 自建内容分发或补丁下发系统
- 把扫描器逻辑整体迁入后端
- 上来就拆微服务
- 引入过重的基础设施，如消息队列或 Elasticsearch 作为前置条件
- 先做复杂的可视化平台外壳

## 9. 成功标准

第一阶段的成功不以“功能数量”衡量，而以闭环质量衡量。

建议的成功标准是：

- 能完成一次扫描结果导入并形成可查询记录
- 能查看扫描任务、问题列表和问题详情
- 能完成至少一条完整的问题状态流转
- 能保留与扫描相关的附件或报告信息
- 能通过最小统计视图看到问题变化趋势

## 10. 当前默认假设

在你还没有正式拍板前，当前先采用以下默认假设：

- 第一可运行版本优先做质量门禁闭环
- 第一可运行版本优先做 API 和导入链路，再决定是否同步补最小 Web 页面
- 架构采用 Go 模块化单体
- 数据存储优先 PostgreSQL，对象产物优先 MinIO / S3

## 11. 待确认问题

以下问题尚未最终固定，后续会进入 `docs/owner_decisions/`：

- 第一版是否严格收敛到质量门禁闭环，还是同步落一部分发布控制
- 第一可运行交付是 API 优先，还是直接带最小管理页面
- 问题指纹的第一版生成策略怎么定
- 附件和报告的最小保留策略怎么定
- 统计报表第一版做到什么深度

## 12. 结论

内容控制平台的第一目标，不是做一个“大而全的后台”，而是先做出一个边界清楚、状态可追踪、问题可闭环、后续可扩展到发布治理的控制面。

后续所有设计和实现，都应优先服务这个目标。
