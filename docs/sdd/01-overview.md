# 01. Overview

## 1. 文档目的

本文档用于定义内容控制平台第一阶段的系统总览、边界、目标闭环和模块基线。

它回答的是：

- 这个系统从系统设计角度到底是什么
- 第一阶段先交付哪段闭环
- 模块边界应该怎样切
- 后续 SDD 主题之间如何协同

## 2. 系统定义

内容控制平台是一个面向游戏研发生产链路的内部控制面。

它不直接承担 Unity 扫描器、资源分发网络或实时游戏逻辑服职责，而是负责把扫描结果、问题状态、资源归属、统计视图和后续发布治理收敛到统一后端中。

从系统视角看，它的本质是：

- 一个结果接入和治理平台
- 一个问题状态与责任追踪平台
- 一个后续可扩展到发布控制的控制面后端

## 3. 系统边界

### 3.1 系统负责什么

第一阶段系统负责：

- 接收扫描结果和相关附件
- 记录扫描任务和来源元信息
- 建立资源、规则、问题和责任归属关系
- 提供问题查询、详情和基础状态流转能力
- 提供最小统计和后续治理所需的可追踪基础

### 3.2 系统暂时不负责什么

第一阶段系统不负责：

- Unity 扫描器规则执行本身
- 资源 CDN、补丁分发和下载链路
- 实时游戏逻辑服务
- 高可用集群和复杂运维平台
- 过重的多服务拆分

### 3.3 外部依赖边界

系统与外部边界目前是：

- 上游：Unity 检查器、命令行扫描器、CI
- 存储：PostgreSQL、对象存储
- 未来外部能力：通知、定时任务、审批和发布系统集成

## 4. 当前阶段目标

### 4.1 第一阶段目标

第一阶段不是做“大而全的平台外壳”，而是先交付质量门禁控制面的最小闭环。

默认推荐闭环为：

1. 导入扫描结果
2. 查询扫描任务
3. 查询问题列表
4. 查看问题详情
5. 执行问题状态流转

### 4.2 当前成功标准

第一阶段设计应服务这些可验证目标：

- 能导入一份有效扫描结果
- 能形成可查询的扫描任务记录
- 能基于项目、规则、资源、责任人和状态筛选问题
- 能查看问题详情及其关联上下文
- 能完成至少一条问题状态变化并留下操作记录

## 5. 当前默认实现假设

在正式拍板前，当前设计按以下默认方案推进：

- 第一版收敛到质量门禁闭环
- 第一可运行版本优先 API 和导入链路
- 架构形态采用 Go 模块化单体
- Web 层使用 `Gin`
- 数据存储采用 PostgreSQL + MinIO / S3

如果这些默认方案之后被推翻，应同步更新 `product_intent`、`roadmap`、`backlog`、`owner_decisions` 和相关 SDD。

## 6. 架构基线

### 6.1 总体架构风格

第一版采用模块化单体。

原因不是因为系统永远不会拆，而是因为当前最核心的问题仍然是：

- 幂等导入
- 问题指纹
- 状态流转
- 查询过滤
- 审计与可追踪

这些问题在第一阶段更依赖一致的数据模型和清晰的模块边界，而不是更细的进程拆分。

### 6.2 模块基线

当前模块基线为：

- 结果接入模块
- 扫描任务模块
- 问题管理模块
- 资源索引模块
- 状态流转模块
- 统计报表模块
- 发布控制模块
- 权限与用户模块
- 审计模块
- 通知与异步任务模块

其中第一阶段真正进入实现优先级前列的是：

- 结果接入
- 扫描任务
- 问题管理
- 资源索引
- 状态流转

统计、通知和发布控制在第一版设计中保留位置，但不应抢占最小闭环的实现顺序。

## 7. 分层约束

### 7.1 Web 层

Web 层使用 `Gin`，只负责：

- 路由
- 参数绑定
- 中间件
- HTTP 响应输出

不负责承载核心业务逻辑。

### 7.2 应用层

应用层负责：

- 用例编排
- 事务边界
- 导入流程协调
- 查询与状态流转的应用规则

### 7.3 领域层

领域层负责：

- 问题对象和状态模型
- 问题指纹规则
- 扫描任务和资源对象的业务语义
- 核心约束和状态变化规则

### 7.4 基础设施层

基础设施层负责：

- PostgreSQL 持久化
- 对象存储接入
- 日志
- 迁移
- 通知和调度的技术实现

## 8. 当前核心系统流

当前系统的最小流转可概括为：

1. 上游工具生成扫描结果。
2. 结果接入模块接收输入和附件。
3. 系统创建或识别扫描任务。
4. 系统归一化资源、规则和问题信息。
5. 问题进入可查询状态，并关联资源、扫描任务和附件。
6. 用户通过 API 查询并执行状态流转。
7. 系统保留问题动作日志，为统计和审计提供基础。

## 9. 第一阶段设计重点

当前阶段最值钱的设计点不在“框架是否高级”，而在：

- 导入是否可幂等
- 问题是否可稳定识别
- 资源和问题的归属关系是否清楚
- 状态流转是否有历史
- 查询和统计维度是否能支撑治理
- 失败场景是否可恢复

## 11. 当前结论

内容控制平台第一阶段应该被实现成一个边界清楚、模块清楚、运行约束明确的控制面后端。

这意味着第一阶段设计要优先服务最小闭环，而不是提前优化未来所有可能的扩展方向。
