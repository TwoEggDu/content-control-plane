# 02. Ingestion and Issue Domain

## 1. 文档目的

本文档用于定义第一阶段最核心的两部分设计：

- 扫描结果接入链路
- 问题领域模型

它回答的是：

- 扫描结果怎样进入系统
- 系统怎样把输入归一化成可治理的问题对象
- 问题指纹第一版怎样定义才足够稳定
- 导入失败和重复导入怎样处理

## 2. 设计目标

第一阶段接入和问题域设计需要同时满足这些目标：

- 导入可追溯
- 导入可幂等
- 问题可识别
- 问题可流转
- 问题可统计
- 失败可恢复

如果做不到这些，后续的查询、统计、通知和发布治理都会建立在不稳定基础上。

## 3. 输入边界

### 3.1 上游输入来源

第一阶段允许的上游来源包括：

- Unity 检查器
- 命令行扫描器
- CI 任务

### 3.2 输入类型

结果接入模块至少应支持这些输入：

- 扫描结果 JSON
- 报告文件
- 日志文件
- 截图或其他附件
- 扫描任务元信息

### 3.3 当前不承担的输入职责

结果接入模块不负责：

- 在后台执行扫描规则
- 动态调度 Unity 扫描进程
- 解释所有上游工具的复杂内部逻辑

第一阶段只关心“如何稳定接入结果”，而不是“如何统一所有扫描器实现”。

## 4. 导入链路总览

第一版导入链路建议按下面顺序执行：

1. 接收请求和基础元信息
2. 做输入校验
3. 生成或识别导入幂等键
4. 创建或复用扫描任务记录
5. 解析资源、规则和问题数据
6. 计算问题指纹
7. 写入问题和附件元信息
8. 更新扫描任务统计和最终状态

这条链路里最关键的两个控制点是：

- 幂等判定
- 问题指纹稳定性

## 5. 扫描任务模型

### 5.1 `scan_task` 的职责

`scan_task` 代表一次结果导入批次，而不是一个长期问题实体。

它至少应承载：

- 项目标识
- 来源类型
- 分支名
- 提交号
- 扫描器版本
- 触发者或触发来源
- 开始和结束时间
- 导入状态
- 问题总数

### 5.2 扫描任务状态建议

第一版可先采用这些状态：

- `RECEIVED`
- `PROCESSING`
- `IMPORTED`
- `FAILED`

如果后续需要更细粒度重试或部分成功状态，再扩展。

## 6. 导入幂等策略

### 6.1 为什么必须幂等

扫描结果导入是第一阶段最容易出现重复提交和补偿重试的链路。

如果没有幂等策略，会直接导致：

- 扫描任务重复
- 问题重复计数
- 趋势失真
- 人工处理成本增加

### 6.2 第一版幂等键建议

第一版优先使用以下顺序识别幂等：

1. 上游显式提供的 `task_no` 或等价外部任务号
2. 如果没有外部任务号，则使用组合键，例如：
   - `project`
   - `source_type`
   - `branch_name`
   - `commit_sha`
   - `scanner_version`
   - 输入主文件内容摘要

### 6.3 幂等处理原则

- 如果命中已成功导入任务，应返回已有任务信息，而不是再次写入
- 如果命中失败任务，应允许显式重试，但不能静默覆盖历史失败记录
- 幂等判断必须早于问题明细大批量写入

## 7. 资源域模型

### 7.1 `resource_item` 的定位

`resource_item` 表示被治理的资源对象，不是一次扫描中的临时记录。

它应尽量稳定承载：

- 资源 GUID
- 资源路径
- 资源名称
- 资源类型
- 所属模块
- 责任人

### 7.2 资源识别优先级

第一版建议按以下顺序识别资源：

1. `resource_guid`
2. 规范化后的 `resource_path`
3. 如果都缺失，则降级为导入失败或孤立记录，不进入正常治理流

原因是路径会变，但 GUID 更接近稳定身份。

### 7.3 资源归属原则

- 资源对象应尽量在多次扫描之间复用
- 问题统计优先基于资源对象，而不是基于原始输入文本
- 责任归属优先挂在资源上，问题上允许有覆盖值

## 8. 问题域模型

### 8.1 `issue_item` 的定位

第一版默认把 `issue_item` 视为“可持续治理的问题实体”，而不是每次扫描都新生成一条完全独立的问题快照。

这意味着：

- 一个问题应围绕稳定指纹跨多次扫描延续
- 问题本身拥有状态、责任人和历史动作
- `first_seen_at / last_seen_at / resolved_at` 才有明确业务意义

### 8.2 与当前草图的关系

当前架构草图里 `issue_item` 仍保留了 `scan_task_id` 字段。

第一版设计建议把它解释为：

- 当前或最近一次观察到该问题的扫描任务引用

如果后续需要完整保留“每次扫描中的每次观察结果”，可在第二阶段引入 `issue_occurrence` 或类似观测表，而不是把第一版问题模型做得过重。

### 8.3 `issue_item` 应至少承载的语义

- 问题归属项目
- 问题归属资源
- 规则编码或规则引用
- 严重级别
- 当前状态
- 责任人
- 指纹
- 消息摘要
- 首次发现时间
- 最近发现时间
- 解决时间

## 9. 问题状态模型

第一版问题状态建议采用：

- `NEW`
- `ASSIGNED`
- `FIXING`
- `RESOLVED`
- `VERIFIED`
- `IGNORED`

含义约束：

- `NEW`：系统首次识别到该问题
- `ASSIGNED`：问题已有明确责任人
- `FIXING`：责任人开始处理
- `RESOLVED`：宣称已修复，等待后续扫描验证
- `VERIFIED`：后续扫描确认问题消失
- `IGNORED`：误报、豁免或暂不处理

## 10. 问题指纹策略

### 10.1 指纹目标

第一版问题指纹的目标不是“理论最完美”，而是“足够稳定、足够可解释、足够支持治理”。

好的指纹至少应满足：

- 同一个语义问题在多次扫描中尽量一致
- 不同问题尽量不要碰撞
- 能解释指纹由哪些业务字段组成

### 10.2 第一版指纹输入建议

第一版建议用规范化后的业务元组构造指纹源：

- `project_code`
- `rule_code`
- 资源稳定标识：优先 `resource_guid`，否则 `normalized_resource_path`
- 位置键：例如组件路径、子对象路径、属性路径、检查节点标识
- 问题特征摘要：例如规范化后的 `expected_value/current_value` 或确定性 message key

### 10.3 不应进入指纹的字段

第一版不建议把这些字段放入指纹：

- 扫描时间
- 分支名
- 提交时间
- 责任人
- 当前状态
- 是否已忽略
- 会频繁变化但不改变问题语义的描述文本

### 10.4 指纹生成方式

第一版建议：

1. 先生成规范化字符串
2. 再做稳定哈希，例如 `sha256`
3. 存储最终哈希值，并保留足够的原始业务字段以便排障

### 10.5 当前指纹策略的风险

第一版最大风险不是哈希算法，而是上游字段不稳定。

因此实现时必须允许：

- 记录指纹生成输入的关键字段
- 对疑似碰撞或漂移案例做排查
- 在第二阶段调整位置键或 message key 规则

## 11. 问题与扫描结果的关系

### 11.1 当前处理原则

当同一指纹在新扫描中再次出现时：

- 不应创建一个全新的长期问题实体
- 应更新该问题的 `last_seen_at`
- 可更新最近一次扫描任务引用
- 如问题之前处于 `RESOLVED`，应允许它重新回到开放状态

### 11.2 当问题在新扫描中消失时

第一版建议：

- 不在导入时直接自动改成 `VERIFIED`，除非明确执行了验证逻辑
- 至少要支持后续基于扫描结果做验证判断

这样可以避免把“本次未出现”过早等同于“已确认解决”。

## 12. 附件与原始输入

### 12.1 附件归属

第一版附件优先归属 `scan_task`，因为它们通常描述的是一个扫描批次，而不是单个问题。

### 12.2 原始输入保留策略

建议至少保留：

- 原始输入摘要信息
- 主输入文件对象存储位置
- 相关日志和报告对象存储位置

这样后续才能排查：

- 为什么导入失败
- 为什么指纹和预期不同
- 为什么某条问题在某次扫描中出现或消失

## 13. 第一阶段实现约束

这份设计对后续实现提出这些直接约束：

- 导入接口必须在大批量写库前完成幂等判定
- 资源识别必须优先使用稳定标识
- 问题模型必须支持跨扫描持续治理
- 状态流转必须记录动作日志
- 指纹生成逻辑必须集中管理，不能散落在多个 Handler 中

## 14. 当前开放问题

虽然本文档已经给出默认方案，但这些点后续仍可能进入决策队列：

- 是否在第二阶段增加 `issue_occurrence` 观测表
- 指纹位置键的最小必填字段是什么
- 资源责任人和问题责任人冲突时谁优先
- 验证状态是否依赖专门的验证扫描逻辑

## 15. 当前结论

第一阶段最关键的不是把导入接口“做通”，而是把导入、幂等、资源识别、问题指纹和问题持续治理模型设计稳。

如果这部分设计不稳，后面的查询、统计、通知和发布控制都会建立在不可信的数据基础上。
