# 05. Runtime, Storage, and Ops

## 1. 文档目的

本文档用于定义内容控制平台在当前阶段的运行方式、存储策略和最小运维基线。

它回答的不是“怎么建设完整运维体系”，而是：

- 第一版系统怎样才能稳定跑起来
- 当前必须考虑哪些运行约束
- 出错后如何做到可观察、可排查、可恢复

## 2. 当前结论

当前阶段需要考虑运维，但只考虑最小必要集合，不做过重建设。

换句话说，第一版追求的是：

- 能启动
- 出错能查
- 失败能补

而不是：

- 高可用集群
- 多节点扩缩容
- 完整监控平台
- 复杂告警系统
- 生产级自动化运维体系

## 3. 当前范围与非目标

### 3.1 当前范围

第一版最小运维基线应覆盖：

- 运行模式
- 配置注入
- 数据库迁移
- 对象存储接入
- 日志和审计的最小策略
- 健康检查
- 导入失败后的恢复路径
- 本地启动和基本排障方式

### 3.2 当前非目标

当前阶段不默认建设：

- Kubernetes
- 多副本部署
- Prometheus/Grafana 全套监控
- 复杂链路追踪系统
- 自动扩缩容
- 多环境复杂发布编排

## 4. 第一版运行模型

### 4.1 部署形态

第一版默认采用单体服务运行：

- 一个 `api` 进程提供 HTTP API
- 一个 PostgreSQL 实例提供关系数据存储
- 一个对象存储实例承接附件与报告
- 轻量后台任务优先使用进程内任务或独立 `worker`，但不强依赖消息队列

### 4.2 进程职责

#### `api`

负责：

- 接收导入请求
- 提供查询 API
- 处理状态流转
- 输出健康检查
- 记录请求和错误日志

#### `worker`（可选，后续）

负责：

- 异步报告处理
- 定时任务
- 通知发送
- 重试型后台任务

当前阶段允许先不独立部署 `worker`，但在设计时要保留可拆分空间。

## 5. 配置策略

### 5.1 配置注入原则

第一版采用环境变量或环境变量驱动的配置文件。

必须外置的配置至少包括：

- `APP_ENV`
- `APP_PORT`
- `APP_TIMEZONE`
- `DATABASE_URL`
- `STORAGE_ENDPOINT`
- `STORAGE_BUCKET`
- `STORAGE_ACCESS_KEY`
- `STORAGE_SECRET_KEY`
- `LOG_LEVEL`

### 5.2 配置原则

- 不把环境差异硬编码到代码里
- 不把密钥写入仓库
- 本地开发允许使用 `.env` 类方式，但正式环境仍应由外部注入
- 配置加载失败时，服务应明确报错并拒绝启动

## 6. 存储策略

### 6.1 PostgreSQL

PostgreSQL 用于存储：

- 项目
- 扫描任务
- 资源对象
- 问题对象
- 问题操作日志
- 附件元信息

不用于存储：

- 大体积报告文件
- 截图二进制内容
- 原始扫描产物文件本体

### 6.2 对象存储

对象存储用于保存：

- 扫描原始 JSON
- 检查报告
- 截图
- 日志附件

数据库中只保存这些对象的元信息：

- `storage_key`
- `file_type`
- `content_hash`
- `file_name`
- 归属关系

### 6.3 存储设计原则

- 关系数据和文件数据分离
- 附件上传失败必须可追踪
- 元信息写入和对象上传的失败顺序要可恢复
- 第一版允许先采用最直接策略，但必须定义失败补偿路径

## 7. 数据库迁移策略

### 7.1 迁移工具

第一版采用 `goose` 管理数据库 migration。

### 7.2 迁移要求

- 所有结构变更通过 migration 执行，不直接手改数据库
- 本地和后续环境使用相同迁移入口
- 每次引入表结构变化时，同步更新数据模型文档和相关 SDD

### 7.3 当前最小能力

至少应具备：

- 初始化数据库
- 执行最新 migration
- 回滚最近一批 migration
- 在本地重建开发环境

## 8. 日志、审计与可观察性

### 8.1 当前最小日志要求

第一版不追求完整 observability 平台，但至少要保证关键路径可查。

必须有日志的地方：

- 服务启动与配置加载
- 数据库连接失败
- 导入请求开始和结束
- 幂等冲突或重复导入
- 状态流转失败
- 对象存储上传失败
- 未处理异常

### 8.2 日志字段建议

关键日志至少应包含：

- 时间
- 日志级别
- 请求 ID 或任务 ID
- 项目 ID 或项目编码
- 扫描任务 ID
- 错误摘要

### 8.3 审计边界

业务操作审计和应用日志不是一回事。

- 应用日志：给排障用
- 审计日志：给责任追踪和复盘用

第一版至少要保证问题状态流转可以留下业务动作记录。

## 9. 健康检查与运行探针

### 9.1 最小探针

第一版建议至少提供：

- `GET /healthz`：进程活着
- `GET /readyz`：数据库和关键依赖可用

### 9.2 当前判定原则

- 进程已启动不代表服务已可用
- 如果数据库不可达，`readyz` 应失败
- 如果对象存储是关键依赖，后续可考虑纳入 `readyz`，但第一版可先保守处理

## 10. 失败恢复与补偿

### 10.1 导入链路必须可补

导入是第一版最关键的运维风险点。

必须提前考虑的失败场景：

- 请求接收成功，但数据库写入失败
- 数据库写入成功，但对象存储上传失败
- 同一个扫描结果被重复提交
- 扫描任务创建成功，但问题明细写入中断

### 10.2 当前恢复原则

- 导入流程必须有幂等标识
- 失败状态要能被明确看见，而不是静默丢失
- 允许人工或后台任务重试
- 对于部分成功场景，要定义补偿或清理策略

## 11. 当前阶段的最小 Runbook 需求

在真正写实现前，至少应准备以下操作手册：

1. 本地启动手册
2. 数据库 migration 手册
3. 导入验证手册
4. 对象存储排障手册

这些 Runbook 不是现在就要全部写完，但相关操作在设计时必须可落地。

## 12. 对实现阶段的约束

当前最小运维基线会对后续实现提出这些约束：

- 代码启动时必须集中加载配置
- Handler 不应吞掉关键错误
- 导入流程必须可观测、可重试、可幂等
- 数据库 migration 不能后补，应跟着模型演进同步建设
- 附件元信息和对象存储 key 设计要从第一版就稳定

## 13. 当前结论

对这个项目而言，当下需要考虑运维，但只考虑“最小运维基线”。

第一版不需要重型运维体系，但必须提前把运行方式、配置、日志、迁移、健康检查和失败恢复设计清楚。否则后续最容易返工的不是业务逻辑，而是系统可运行性本身。
