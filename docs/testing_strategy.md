# Testing Strategy

## 文档目的

本文档用于定义这个项目的测试分层、最低验证要求和当前阶段的测试重点。

## 当前判断

这个项目是控制面后端，不适合只靠人工点接口验证。

它的核心风险在：

- 导入幂等是否正确
- 状态流转是否正确
- 查询过滤和统计是否正确
- 数据模型和存储行为是否一致

## 测试分层

### 1. 文档级检查

适用阶段：M0

目标：保证边界、流程和待决策事项一致，不让设计信息漂移。

### 2. 领域层测试

适用阶段：M1 起

目标：验证：

- 问题指纹生成
- 状态流转规则
- 统计聚合的纯逻辑

### 3. 持久化和仓储测试

适用阶段：M1 起

目标：验证：

- 表结构映射
- 查询过滤条件
- 唯一约束和幂等插入
- 事务边界

### 4. API 集成测试

适用阶段：M1 起

目标：验证：

- 导入接口能否正确入库
- 查询接口返回是否符合契约
- 状态流转接口是否同时写入日志

### 5. 端到端冒烟验证

适用阶段：M1 末和之后

目标：验证一条最小业务链：

- 导入扫描结果
- 查询任务
- 查询问题
- 修改状态
- 再查询确认结果

## 当前最小测试要求

在正式实现代码之前，当前先采用这些最小要求：

- 文档结构和入口必须一致
- backlog、roadmap、owner decisions 之间不能互相冲突
- 新增设计默认要说明如何验证

当进入 M1 实现时，建议最低补这些测试：

- 问题状态流转测试
- 问题指纹测试
- 导入幂等测试
- 问题列表过滤测试
- 导入到查询的 API 冒烟测试

## 当前不优先的测试

在第一闭环前，不优先投入：

- 复杂前端 E2E 测试
- 大规模压测
- 完整权限矩阵测试

## 更新时机

满足以下任一条件时更新本文件：

- 增加新的关键模块
- 实现层级从文档设计进入代码阶段
- 测试重点发生变化
- 出现一次因测试缺位导致的明显返工
